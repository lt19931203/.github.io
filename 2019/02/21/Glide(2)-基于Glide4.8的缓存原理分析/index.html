简介

所有用过Glide的朋友都知道，Glide缓存嘛，无非就是分为两部分，内存和磁盘。通过一些API跳过内存缓存，或者改变磁盘缓存的策略，都是老生常谈的东西了，网上的文章也是数不胜数，不过我个人还是喜欢按步骤来拆分源码的。话不多说，直接来看代码。

内存缓存

Glide的内存缓存不止一层，而是两层，分别是ActiveResources（表示当前内存中正在使用的，优先级高）和LruCache（最近最少使用算法，优先级低于ActiveResources），如果有对LruCache算法不了解的，可以在网上搜索了解一下，这里就不介绍了。在上一篇文章中说到了Glide首次加载流程，其中Glide.with().load()这两步都没有做太多的事情，大部分的逻辑都是在into()方法中完成的，我们还认识了一些类，包括他们的方法，做了哪些事…，这篇文章不从头来了，直接从Engine#load()方法开始。因为在Engine#load()之前做的事情，都和缓存关系不大。如果有小伙伴对于Engine#load()来源不清楚的，建议还是先看首次加载流程吧。

步骤9：内存缓存小结

内存缓存分为两层

  1. ActiveResources，当前正在使用的Resouces集合，使用优先级最高，为弱引用方式，来源于Resource加载成功的的回调，通过acquired变量控制在内存中的引用次数，当acquired为0的时候，会被放入LruResourceCache，并在ActiveResources中移除。
  2. LruResourceCache，最近最少使用的Resouces集合，优先级低于ActiveResources，强引用方式，当ActiveResources中有Resource被移除后，会被放入LruResourceCache，如果有Resource被使用了，那么会从LruResourceCache被移除，加入到ActiveResources中

磁盘缓存

说到磁盘缓存不得不说到DiskCacheStrategy，它代表着磁盘缓存策略，它在以前的版本有4个可选项，分别是

  1. DiskCacheStrategy.NONE： 表示不缓存任何内容
  2. DiskCacheStrategy.SOURCE： 表示只缓存原始图片。
  3. DiskCacheStrategy.RESULT： 表示只缓存转换过后的图片（默认选项）。
  4. DiskCacheStrategy.ALL ： 表示既缓存原始图片，也缓存转换过后的图片。

而现在的选项是

  1. DiskCacheStrategy.ALL
  2. DiskCacheStrategy.NONE
  3. DiskCacheStrategy.DATA：表示只缓存原始图片
  4. DiskCacheStrategy.RESOURCE：表示只缓存转换过后的图片
  5. DiskCacheStrategy.AUTOMATIC： 表示让Glide根据图片资源智能地选择使用哪一种缓存策略（默认选项）。

因为我们并未指定DiskCacheStrategy，所以默认是AUTOMATIC，之前说到内存缓存有两层，磁盘缓存其实也是有两层的，也可以说有两种，分别是转换之后的图片缓存，和原始图片缓存。我们接着往下看，假设现在从内存中没有获取到结果，Glide会怎么做。

来看一下这个判断条件，(isFromAlternateCacheKey && dataSource == DataSource.DATA_DISK_CACHE)第一组，与dataSource == DataSource.LOCAL是二选一，另一个encodeStrategy == EncodeStrategy.TRANSFORMED条件是必须要满足，在我们此次的分析中，传入的这三个参数分别是什么呢？

最终的判断结果是false，导致没有进行ResourceCache，只因为isFromAlternateCacheKey这个条件没有满足，而这个条件是由步骤18第24行!decodeHelper.isSourceKey(currentSourceKey)产生的，直接由currentSourceKey决定，而这个decodeHelper.isSourceKey()方法，看名字就知道，用来判断传入的key是不是SourceKey，下面来看一下currentSourceKey赋值的地方。

步骤20：DecodeJob#onDataFetcherReady()

第3行，我们找到了currentSourceKey赋值的地方，而这个onDataFetcherReady()方法也是我们熟悉的方法，在上篇文章步骤59出现过，调用时机为Resource加载成功的回调，继续跟踪这个方法的调用处的话，会发现有4个调用处

    这几个回调方法，返回的第4个参数，dataSource其实也是和步骤19的第二个判断条件相关，和sourceKey一样，也是在这个方法中传入的。现在凭空看这些个回调方法，貌似是没有什么收获的，但我们首先可以排除第二个回调，也就是ResourceCacheGenerator的回调，因为他的dataSource是RESOURCE_DISK_CACHE，并不符合步骤19的条件，第3个和第4个也可以排除了，因为3和4都是从网络上下载原始数据，这一步我们已经执行过了，最后也没有缓存ResourceCache。那就把目光放在DataCacheGenerator上吧，这个DataCacheGenerator#onDataReady()方法就是我们刚刚分析过的原图缓存回调，然而很遗憾，DataCacheGenerator#onDataReady()中的sourceKey是originalKey（可以理解为URL），也就是说，从原图缓存拿到的结果，不会被缓存到ResourceCache中。

步骤21：AUTOMATIC缓存机制

这里给一下解释，当我们使用默认的缓存策略AUTOMATIC时，内部会是什么样的逻辑，这里直接引用Glide官方中文翻译

默认的策略叫做 AUTOMATIC ，它会尝试对本地和远程图片使用最佳的策略。当你加载远程数据（比如，从URL下载）时，AUTOMATIC 策略仅会存储未被你的加载过程修改过(比如，变换，裁剪–译者注)的原始数据，因为下载远程数据相比调整磁盘上已经存在的数据要昂贵得多。对于本地数据，AUTOMATIC 策略则会仅存储变换过的缩略图，因为即使你需要再次生成另一个尺寸或类型的图片，取回原始数据也很容易。

从这段文字得知，当我们采用网络请求获得结果时，只会对原图进行缓存，当加载本地数据（手机上存储的图片）的时候，才会缓存变换过后的ResourceCache。所以，这里已经得到结果了，默认的磁盘策略并不会为我们缓存转换后的图片的。不过我们之前的分析也没有白费，通过分析AUTOMATIC，也能帮助我们了解其他缓存策略是如何工作的。

步骤22：加载本地图片

当我们首次调用上方代码，缓存策略同样是AUTOMATIC，最终在步骤19中传入的参数就会是

  1. isFromAlternateCacheKey = false;
  2. dataSource = DataSource.LOCAL;
  3. encodeStrategy = EncodeStrategy.TRANSFORMED;

第二个参数变成了DataSource.LOCAL，从而满足了步骤19的条件，进而进行ResourceCache的缓存
